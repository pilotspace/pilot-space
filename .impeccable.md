## Design Context

### Users
Software development teams (5-100 members per workspace) using Pilot Space for AI-augmented SDLC. Users range from senior engineers to product managers and designers. They interact in a "Note-First" workflow: brainstorm in a rich text canvas, then structure ideas into issues, cycles, and PRs. The AI co-pilot assists with ghost text completions, issue extraction, PR reviews, and context-aware suggestions. Users bring their own API keys (BYOK model) and need to feel the AI is a teammate, not a black box.

### Brand Personality
**Warm, Capable, Collaborative** — confirmed as the three defining adjectives.

- **Voice**: Approachable and knowledgeable. The AI speaks as a helpful teammate, never a system notification. Guidance, not commands.
- **Tone**: Confident but not arrogant. Friendly but not casual. Professional but not sterile.
- **Emotional goals**: Primarily calm confidence — users feel in control and supported. Secondary layers of professional trust (the platform is reliable), creative energy (it sparks momentum), and playful delight (micro-interactions that surprise and charm).

### Aesthetic Direction

**Visual tone**: Pilot Space v2 blends three design influences into a cohesive SDLC command center:

| Influence | What it contributes |
|-----------|-------------------|
| **Cohere structure** | 22px signature radius on content cards, enterprise spacing, serif/sans hierarchy |
| **Gemini interaction** | Borderless blue-gray chatbox surface, bare icon toolbar, outlined tool pills, right-aligned mode selector |
| **Pilot Space warmth** | Colored gradient card tops (green/blue/purple), brand green accent, warm messenger-source avatars, status-semantic color coding |

**References**:

| Product | What we take |
|---------|--------------|
| Craft | Rich layered surfaces, beautiful typography, calm sophistication |
| Apple | Squircle corners, frosted glass, tinted shadows, material depth |
| Things 3 | Natural color accents, spacious calm, delightful minimalism |
| Linear | Clean information hierarchy, keyboard-first interactions |
| Cohere | Enterprise card structure, 22px radius language, serif/sans pairing |
| Gemini | Borderless AI chatbox, toolbar pill pattern, blue-gray surface |

**Anti-references** (what Pilot Space must NOT be):
- Generic SaaS defaults (Inter font, orange accent, pure white, cookie-cutter shadcn)
- Dense enterprise tools (Jira, ServiceNow — information overload, tiny text, gray-on-gray)
- Flashy/trendy startups (gratuitous gradients, glass effects, animations over substance)
- Developer-only CLI aesthetic (must feel approachable to non-engineers on the team)

**Theme**: Light + dark mode, both fully implemented. Light is warm off-white (#ffffff base, #f7f7f5 subtle). Dark is soft dark (#191919 base, #222222 subtle).

**Typography**:

| Role | Font | Fallbacks | Usage |
|------|------|-----------|-------|
| Display | `Fraunces` | Georgia, serif | Hero greeting ONLY, 24px, weight 400, -1px tracking |
| Body / UI | `Inter` | system-ui, -apple-system, sans-serif | All functional text |
| Code / Labels | `JetBrains Mono` | monospace | Artifact badges, timestamps, status text |
| Icons | Lucide React | — | 1.5px stroke, rounded caps |

Key type scale:
- Hero greeting: Fraunces 24px/400 (single line, compact)
- Section heading: Inter 16px/600
- Card title: Inter 13px/500-600
- Toolbar label: Inter 12-13px/500
- Input placeholder: Inter 16px/normal
- Artifact badge: JetBrains Mono 10px/600, letter-spacing 0.4px
- Detail line: Inter 11px/normal
- Status badge: Inter 10px/600

**Spacing**: 4px grid. Generous whitespace — spacious layouts are a feature, not wasted space.

Key vertical rhythm (homepage):
- Pill → Greeting: 36px
- Greeting → Chatbox: 36px
- Chatbox → Artifacts: 80px (20+40+40 consolidated spacers)
- Between content sections: 16px internal padding
- Artifact card grid gap: 16px
- Routine/sprint card gap: 12px

**Border radius scale**:

| Scale | Value | Usage |
|-------|-------|-------|
| Sharp | 4px | Project badges in artifact cards |
| Comfortable | 6px | Artifact type badge corners |
| Card | 16px | Routine task cards, sprint progress rows |
| **Signature** | **22px** | Artifact cards — the defining content radius |
| **Chatbox** | **28px** | AI Prompt Hero container (Gemini-scale) |
| Pill | 9999px | Workspace pill, tool pills, status badges, send button |

**Depth & elevation** (no arbitrary shadows):

| Level | Treatment | Usage |
|-------|-----------|-------|
| L0 Flat | No shadow, no border | Page background, text blocks |
| L1 Surface | Colored fill, no border | AI Prompt Hero (`#f0f4f9`) |
| L2 Bordered | 1px solid `#e5e7eb` | Artifact cards, routine cards, sprint rows |
| L3 Gradient | Directional gradient fill | Artifact card top sections |

Only the sidebar right border provides visual separation. All content cards use border containment. The chatbox uses fill-color contrast (blue-gray on white), never borders or shadows.

**Animation**: CSS transitions + Framer Motion. Minimal and functional — animations serve UX, never decoration. Respect `prefers-reduced-motion`.

### v2 Design Rules

**Do:**
- Use 28px radius on the chatbox, 22px on artifact cards, 16px on list-style cards
- Use Fraunces for the hero greeting ONLY at 24px — nowhere else
- Keep messenger source circles at 34px with saturated brand colors
- Use outlined pills (1px `#c8d1db`) for toolbar items in the chatbox
- Show status through both color AND text (accessibility)
- Use inline detail strings for compact metadata: `"78% complete · Frontend v2 · via @tin · 4m ago"`
- Keep artifact card gradients warm (green/blue/purple) — this is the visual personality
- Use `#f0f4f9` chatbox surface for AI input zones — Gemini blue-gray, no borders

**Don't:**
- Don't add borders or shadows to the chatbox — it uses fill-color contrast only
- Don't mix messenger colors with status colors (source = avatar, status = badge)
- Don't use more than 3 artifact cards in the grid (content density limit)
- Don't skip the source circle on routine cards — it's the primary identifier
- Don't use 22px radius on routine/sprint cards — 16px maintains hierarchy
- Don't add weight 700+ on any text
- Don't reduce routine card gap below 12px
- Don't use arbitrary shadows — the depth system is L0-L3 only

### Responsive Scaling

| Breakpoint | Key Changes |
|-----------|-------------|
| < 768px | Sidebar collapses; chatbox full-width; artifact cards stack 1-column; chatbox 28px → 22px |
| < 640px | Hero scales to 20px; routine cards show source + title only (hide detail) |
| < 425px | Sprint bars hide (show % text); chatbox toolbar wraps; artifact 22px → 16px; routine 16px → 12px |

### Accessibility Requirements

- **Baseline**: WCAG 2.2 AA (mandatory) with aspirational AAA target
- **Color blindness**: All state and priority colors must work for deuteranopia, protanopia, and tritanopia — always pair color with icons/patterns
- **Screen readers**: ARIA live regions for dynamic content, landmark roles for navigation, structured announcements for complex interactions (TipTap editor, drag-and-drop, SSE streaming updates)
- **Keyboard**: Full keyboard navigation, visible focus indicators, skip-to-content links
- **Motion**: Respect `prefers-reduced-motion` — disable non-essential animations
- **Contrast**: 4.5:1 minimum (AA), accessible text color tokens already defined (`--primary-text: #1F7D66`, `--ai-text: #4A6F8F`)

### Design Principles

1. **Think first, structure later** — The interface encourages free-form thinking before imposing structure. Notes are the entry point, not forms.
2. **AI is a teammate, not a toolbar** — AI suggestions appear naturally in the workflow with clear "You + AI" attribution. Never hide the AI or make it feel bolted on.
3. **Spacious calm over dense efficiency** — Generous whitespace and clear hierarchy. Progressive disclosure over showing everything at once. Breathing room is a feature.
4. **Warmth through craft** — Every surface, shadow, and transition should feel intentional and human. No default values. No generic UI. Details matter.
5. **Accessible by default** — Accessibility is not an afterthought. Color alone never conveys meaning. Keyboard paths exist for every interaction. Screen reader experience is first-class.
