    **Schema changes** (`backend/src/pilot_space/api/v1/schemas/transcription.py`):
    - Add `audio_url: str | None = None` field to `TranscribeResponse`. This holds a signed URL for audio playback. It is `None` when storage upload fails (non-blocking) or for cached responses where audio was already stored.
    - Add `audio_storage_key: str | None = None` field to `TranscribeResponse`. This is the raw storage key for future reference (e.g., to generate fresh signed URLs).

    **Endpoint changes** (`backend/src/pilot_space/api/v1/routers/transcription.py`):
    After the existing ElevenLabs transcription succeeds AND the cache upsert completes, add audio storage:

    1. Import `SupabaseStorageClient` from `pilot_space.infrastructure.storage.client` and `StorageUploadError` for error handling.
    2. Use bucket name `"voice-recordings"` (separate from `"note-artifacts"` — different lifecycle, no project scope).
    3. Construct storage key: `f"{workspace_id}/{user_id}/{record_id}.webm"` — scoped by workspace and user, using the transcript cache record ID as unique identifier.
    4. Wrap the upload in try/except: if `StorageUploadError` occurs, log warning and continue — audio storage failure MUST NOT block transcription result. Set `audio_url = None` on failure.
    5. On success, generate a signed URL (1 hour expiry) using `storage_client.get_signed_url()`. Set `audio_url` to the signed URL and `audio_storage_key` to the storage key.
    6. For cache hits (cached=True), skip upload (audio was already stored on original transcription). Set `audio_url = None` and `audio_storage_key = cached.storage_key` if the cached record has a storage key. Note: the TranscriptCache model does not have a storage_key column yet — for cached responses, just return `audio_url=None, audio_storage_key=None`. A future task can add storage_key to the cache model.
    7. Include `audio_url` and `audio_storage_key` in both cache-hit and fresh TranscribeResponse returns.

    Do NOT modify the TranscriptCache model — audio storage is best-effort and the cache already works fine without it. The storage key is ephemeral (derivable from workspace_id/user_id/record_id).

    Do NOT add the `voice-recordings` bucket creation — Supabase Storage buckets are created via dashboard or migration. Add a code comment noting the bucket must exist.
