Key: populated_db fixture (in security/conftest.py) already sets up users and workspaces but
does NOT create Projects, States, Issues, Notes, or AuditLog entries. The tests must create
these domain objects themselves using service_role (bypasses RLS) via the same db_session that
populated_db used (which already called await db_session.commit()).

IMPORTANT: The db_session fixture in security/conftest.py does NOT wrap in an outer transaction
(it rolls back after yield). Data inserted before set_rls_context is called is visible to the
RLS query only if committed. Use db_session.flush() + set_rls_context + SELECT pattern since
the session is already in an open transaction from populated_db.commit().

Actually: populated_db calls await db_session.commit() which commits the users/workspaces data.
Tests receive the SecurityTestContext and the db_session. For test data (Issues, Notes, AuditLog),
insert using db_session.add() + await db_session.flush() (not commit) so data is visible in the
same session but still within the transaction that rolls back after the test.

CRITICAL CONSTRAINT: set_rls_context uses set_config(..., true) which is transaction-local
(third arg = is_local). This means RLS context is active only within the current transaction.
Since db_session is already in an open transaction, calling set_rls_context and then querying
will apply RLS within that same transaction.

**test_cross_workspace_issue_access:**
1. Create Project in workspace_b (owner: ctx.outsider) — need name, identifier, workspace_id
2. Create State in workspace_b linked to that project
3. Create Issue in workspace_b with sequence_id=1, state_id, project_id, reporter_id=ctx.outsider.id, name="WS-B Issue", workspace_id=ctx.workspace_b.id
4. await db_session.flush()  # make visible in current transaction
5. Call set_test_rls_context(db_session, ctx.owner.id, ctx.workspace_a.id)  # from security/conftest.py
6. Execute: result = await db_session.execute(select(Issue).where(Issue.workspace_id == ctx.workspace_b.id))
7. issues = result.scalars().all()
8. assert len(issues) == 0, "RLS must block workspace B issues from workspace A user"

Note: set_test_rls_context uses SET LOCAL (transaction-local). The RLS policy on issues checks
workspace_members using current_setting('app.current_user_id'). Since ctx.owner is only a member
of workspace_a (not workspace_b), the subquery returns only workspace_a's id — workspace_b issue
is filtered out.

**test_cross_workspace_note_access:**
Same pattern:
1. Create Note in workspace_b: workspace_id=ctx.workspace_b.id, owner_id=ctx.outsider.id, title="WS-B Note", content={}

3. Call set_test_rls_context(db_session, ctx.owner.id, ctx.workspace_a.id)
4. Execute: result = await db_session.execute(select(Note).where(Note.workspace_id == ctx.workspace_b.id))
5. assert len(result.scalars().all()) == 0

**test_cross_workspace_audit_log_access:**
