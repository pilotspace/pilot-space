## User Constraints (from CONTEXT.md)

### Locked Decisions
- **OIDC**: Handled natively through Supabase GoTrue's built-in OAuth2/OIDC support. Covers Google Workspace, Azure AD (OIDC mode), Okta (OIDC mode). No custom backend code for the happy path.
- **SAML 2.0**: Handled by FastAPI backend using `python3-saml` (OneLogin). Flow: SP-initiated → IdP → POST assertion to `/api/v1/auth/saml/callback` → backend validates assertion → creates/updates Supabase user via admin API → returns Supabase JWT. Avoids Supabase Enterprise tier.
- **Tested providers**: Okta, Azure AD, Google Workspace (all three end-to-end before phase complete)
- **SSO-only enforcement**: `workspace_settings.sso_required` boolean. When true, email/password login rejected at backend auth middleware for that workspace's members.
- **Role claim mapping**: Admin configures `claim_key` + JSON mapping table in workspace settings. Unmapped → `Member` role.
- **Custom RBAC table**: `custom_role` (id, workspace_id, name, description, permissions JSONB, created_at)
- **Permissions format**: Array of `"resource:action"` strings — resources: `issues`, `notes`, `cycles`, `members`, `settings`, `ai`, `integrations`; actions: `read`, `write`, `delete`, `manage`
- **custom_role assignment**: `workspace_member.custom_role_id` nullable FK. If null, falls back to existing `WorkspaceRole` enum.
- **Session table**: `workspace_session` (id, user_id, workspace_id, session_token_hash SHA-256, ip_address, user_agent, created_at, last_seen_at, revoked_at nullable)
- **Session recording**: Middleware records on first auth; updates `last_seen_at` max once per 60s
- **Force-terminate**: Sets `revoked_at` + deletes Redis key + calls `auth.admin.sign_out(userId)`
- **SCIM endpoint prefix**: `/api/v1/scim/v2/` — no standard auth middleware, uses SCIM bearer token
- **SCIM scope**: Core User endpoints only — GET/POST/PUT/PATCH/DELETE `/Users`, GET `/ServiceProviderConfig`. No Groups in v1.
- **SCIM auth**: Bearer token generated by admin, stored as hash in `workspace_settings.scim_token_hash`
- **SCIM deprovisioning**: `DELETE /Users/{id}` → `workspace_member.is_active=false` + revoke sessions. Data never deleted.

## Summary

Phase 1 covers four distinct subsystems: SSO (SAML + OIDC), custom RBAC, session management, and SCIM provisioning. The existing codebase provides strong foundations — `SupabaseAuth` for JWT handling, `WorkspaceScopedModel` base class for new tables, `BaseRepository` for data access, and the settings feature structure for new admin pages. No subsystem requires fundamental rewrites; all build additively on existing patterns.

The SAML path is the most complex: `python3-saml` (v1.16.0, requires system-level `libxmlsec1-dev`) validates assertions and maps users to Supabase via the GoTrue admin API. The self-hosted GoTrue configuration requires `GOTRUE_SAML_ENABLED=true` and `GOTRUE_SAML_PRIVATE_KEY` env vars, plus exposing the ACS and metadata endpoints through Kong. OIDC for Google/Azure/Okta routes entirely through Supabase's existing OAuth2 provider support with no new backend auth code — only workspace-level configuration storage.

SCIM 2.0 is a straightforward FastAPI router implementing RFC 7644's User resource endpoints. The `scim2-models` library (v0.6.4, Pydantic v2) handles request/response serialization. Pagination follows the standard startIndex/count pattern. The main complexity is the separate bearer token authentication that bypasses the standard JWT middleware.

**Primary recommendation:** Implement in four independent tracks (SAML, OIDC, RBAC, Sessions+SCIM) with a shared Wave 0 database migration. Each track produces backend service + router + frontend settings page as a unit.

### Pattern 1: SAML SP-Initiated Flow

**What:** Backend acts as Service Provider; frontend redirects user to IdP; IdP posts assertion back.

**When to use:** When workspace has SAML provider configured.

**Flow:**
```
Frontend GET /api/v1/auth/saml/sso?workspace_id=X
  → Backend: SamlAuth(settings).login() → returns IdP redirect URL
  → Frontend redirects user to IdP

IdP POST /api/v1/auth/saml/callback (form data: SAMLResponse)
  → Backend: SamlAuth.process_response() → validates signature, extracts attributes
  → Backend: maps claims to role via workspace settings
  → Backend: supabase.auth.admin.create_user() or update_user_by_id()
  → Backend: issues Supabase JWT via admin sign-in, returns to frontend
```

**Implementation notes:**
- `python3-saml` requires `settings.json` with SP and IdP metadata
- SP metadata available at `GET /api/v1/auth/saml/metadata`
- GoTrue must have `GOTRUE_SAML_ENABLED=true` + `GOTRUE_SAML_PRIVATE_KEY`
- ACS URL registered with IdP: `https://{host}/api/v1/auth/saml/callback`

### Pattern 2: OIDC SSO via Supabase GoTrue

**What:** Workspace admin configures Okta/Azure AD/Google Workspace OIDC; users click "Continue with SSO"; Supabase GoTrue handles the entire OAuth2 PKCE flow.

**When to use:** OIDC-mode providers (all three target providers support this).

**Flow:**
```
Frontend: supabase.auth.signInWithOAuth({ provider: 'azure', scopes: 'openid email profile' })
  → Supabase GoTrue redirects to IdP
  → User authenticates → IdP returns to Supabase callback
  → Supabase issues JWT with IdP claims in app_metadata
  → Post-auth webhook or frontend check: extract role claim from JWT
  → Call backend PATCH /workspace/{id}/member/role to apply mapped role
```

**Critical note:** OIDC claim → role mapping must happen at post-login time (frontend reads JWT claims, calls backend to set role). Role claim is in `app_metadata` or `user_metadata` of the Supabase JWT.

### Pattern 3: Custom RBAC Permission Check

**What:** `check_permission(user_id, workspace_id, resource, action)` — resolves effective permissions from either built-in role or custom_role.

**When to use:** Everywhere a resource-specific permission check is needed beyond simple isAdmin().

**Example:**
```python
# Source: new permissions.py module
async def check_permission(
    session: AsyncSession,
    user_id: UUID,
    workspace_id: UUID,
    resource: str,
    action: str,
) -> bool:
    member = await member_repo.get_by_user_workspace(user_id, workspace_id)
    if member is None:
        return False
    if member.custom_role_id:
        role = await custom_role_repo.get(member.custom_role_id)
        return f"{resource}:{action}" in (role.permissions or [])
    # Fall back to built-in role capabilities
    return _builtin_role_can(member.role, resource, action)
```

### Pattern 4: Session Recording Middleware

**What:** FastAPI middleware records `workspace_session` rows on first auth; throttled `last_seen_at` update (max 1/60s per session).

**Redis key schema:** `session:revoked:{workspace_id}:{session_token_hash}` with TTL matching token expiry.

**Revocation check:** Auth middleware checks Redis key existence before processing request. If revoked, return 401.

### Pattern 5: SCIM Bearer Token Auth

**What:** SCIM endpoints use a separate auth dependency that validates Bearer token against `workspace_settings.scim_token_hash` (SHA-256).

**Why separate:** SCIM requests come from IdP systems, not user browsers; no Supabase JWT involved.

```python
# Source: api/v1/routers/scim.py
async def get_scim_workspace(
    request: Request,
    workspace_slug: str,
    session: SessionDep,
) -> Workspace:
    """Validate SCIM Bearer token and return workspace."""
    auth_header = request.headers.get("Authorization", "")
    if not auth_header.startswith("Bearer "):
        raise HTTPException(status_code=401)
    token = auth_header[7:]
    token_hash = hashlib.sha256(token.encode()).hexdigest()
    workspace = await workspace_repo.get_by_slug(workspace_slug, session)
    settings = workspace.settings or {}
    if settings.get("scim_token_hash") != token_hash:
        raise HTTPException(status_code=401)
    return workspace
```

### Anti-Patterns to Avoid
- **Putting SAML logic in `auth.py`**: That file is already large. New `auth_sso.py` router + `saml_auth.py` infrastructure module.
- **Storing SAML private key in DB**: Must be an environment variable (`SAML_PRIVATE_KEY_PEM`), not workspace settings.
- **Blocking SAML validation on I/O**: `python3-saml` validation is CPU-bound (XML parsing); run synchronously, no `await`.
- **Lazy-loading members in loops**: N+1 risk when iterating members to check custom roles. Use `joinedload(WorkspaceMember.custom_role)` in repository queries.
- **Trusting role claims without mapping validation**: Always validate `claim_value` against the configured mapping; reject unmapped values to `Member` (never `Admin`).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SAML assertion parsing & signature validation | Custom XML parser + signature code | `python3-saml` | XML signature validation has 20+ edge cases; replay protection, canonicalization, binding variants |
| SCIM payload validation | Custom Pydantic schemas | `scim2-models` | SCIM PATCH (RFC 7644 §3.5.2) has complex `op`/`path`/`value` semantics; scim2-models handles them |
| User-agent parsing | Regex | `ua-parser[regex]` | UA string format has thousands of variants; the regex DB is maintained by browser vendors |
| SHA-256 token hashing | Custom crypto | `cryptography` (already installed) `hashlib` | Use `hashlib.sha256()` — stdlib, no additional dep needed |

**Key insight:** SAML and SCIM are protocol-heavy specs; any hand-rolled implementation will have security gaps or interoperability failures with real IdPs (Okta, Azure AD have strict requirements).

## Common Pitfalls

### Pitfall 1: xmlsec System Dependency Not in Docker Image
**What goes wrong:** `python3-saml` installs via pip but fails at runtime with `ImportError: libxmlsec1.so not found`.
**Why it happens:** `python-xmlsec` (C extension) requires system-level libraries that aren't in base Python images.
**How to avoid:** Add to Dockerfile: `RUN apt-get install -y libxml2-dev libxmlsec1-dev libxmlsec1`
**Warning signs:** `python3-saml` installs without error but `import onelogin.saml2` fails in container.

### Pitfall 2: SAML ACS URL Mismatch
**What goes wrong:** IdP returns user to wrong URL or validation fails with "ACS URL mismatch" error.
**Why it happens:** The ACS URL registered at the IdP must exactly match what's in the SP `settings.json` (including scheme, port, path).
**How to avoid:** Derive ACS URL from `settings.BACKEND_URL` at runtime, not hardcoded. Test with all three IdPs.
**Warning signs:** Okta/Azure return HTTP 400 after SAML response.

### Pitfall 3: Role Claim Mapping After OIDC Login
**What goes wrong:** User logs in via OIDC, gets `Member` role instead of mapped role.
**Why it happens:** Supabase GoTrue issues the JWT before our backend can intercept claims. The role mapping must happen post-login in the frontend or via a webhook.
**How to avoid:** After OIDC login, frontend reads JWT `app_metadata`/`user_metadata`, calls backend `PATCH /workspaces/{id}/members/{userId}/role` with the mapped role (validated server-side).
**Warning signs:** New SSO users all land as Member regardless of IdP group membership.

### Pitfall 4: SCIM PATCH vs PUT Semantics
**What goes wrong:** `PATCH /Users/{id}` replaces the whole user instead of applying operations.
**Why it happens:** RFC 7644 PATCH uses `Operations` array with `op`/`path`/`value` — not a standard JSON merge patch.
**How to avoid:** Use `scim2-models` `PatchOp` to parse PATCH bodies. Support `add`, `remove`, `replace` operations.
**Warning signs:** Okta SCIM provisioner sends PATCH and user fields get corrupted.

### Pitfall 5: Session last_seen_at Write Storm
**What goes wrong:** High-traffic workspaces generate thousands of DB writes per minute from session updates.
**Why it happens:** Naively updating `last_seen_at` on every request causes heavy DB write load.
**How to avoid:** Track last-update timestamp in Redis (key: `session:lastseen:{token_hash}`). Skip DB update if last update was < 60 seconds ago.
**Warning signs:** `workspace_session` table takes disproportionate DB write load.

### Pitfall 6: RLS on New Tables Uses Hardcoded Role Strings
**What goes wrong:** RLS policies reference `'OWNER'` but existing DB stores lowercase enum values, causing members to be invisible.
**Why it happens:** Known enum case mismatch bug (documented in STATE.md).
**How to avoid:** Verify current DB enum values with `SELECT DISTINCT role FROM workspace_members LIMIT 10` before writing RLS policies. Match policy strings to actual stored values.
**Warning signs:** Admin users cannot see data scoped to new tables.

### Pitfall 7: SCIM Token in Workspace Settings JSONB
**What goes wrong:** SCIM token hash stored inconsistently across settings updates if `settings` JSONB is replaced wholesale.
**Why it happens:** If settings update endpoints do a full replace rather than merge, SCIM token gets wiped.
**How to avoid:** Use `jsonb_set()` or merge strategy in workspace settings updates. Or add `scim_token_hash` as a dedicated column (cleaner).
**Warning signs:** SCIM provisioning breaks after admin changes any workspace setting.

### SAML Auth Provider Class
```python
# infrastructure/auth/saml_auth.py
from onelogin.saml2.auth import OneLogin_Saml2_Auth

class SamlAuthProvider:
    def __init__(self, workspace_id: UUID, settings: dict) -> None:
        self._workspace_id = workspace_id
        self._settings = settings  # SP + IdP config from workspace_settings

    def prepare_request(self, request: Request) -> dict:
        """Convert FastAPI Request to python3-saml request format."""
        return {
            "https": "on" if request.url.scheme == "https" else "off",
            "http_host": request.headers.get("host", ""),
            "script_name": request.url.path,
            "get_data": dict(request.query_params),
            "post_data": {},  # populated in callback handler
        }

    def get_login_url(self, request: Request, return_to: str) -> str:
        """Return IdP redirect URL for SP-initiated login."""
        auth = OneLogin_Saml2_Auth(self.prepare_request(request), self._settings)
        return auth.login(return_to=return_to)

    def process_response(self, request: Request, post_data: dict) -> dict:
        """Validate SAML response, extract user attributes."""
        req = {**self.prepare_request(request), "post_data": post_data}
        auth = OneLogin_Saml2_Auth(req, self._settings)
        auth.process_response()
        if not auth.is_authenticated():
            raise SamlValidationError(auth.get_last_error_reason())
        return {
            "name_id": auth.get_nameid(),
            "attributes": auth.get_attributes(),
        }
```

### Session Recording Middleware (throttled)
```python
# api/v1/middleware/session_recording.py
LAST_SEEN_TTL = 60  # seconds

async def record_session(
    redis: Redis,
    db: AsyncSession,
    token_hash: str,
    user_id: UUID,
    workspace_id: UUID,
    ip: str,
    user_agent: str,
) -> None:
    redis_key = f"session:lastseen:{token_hash}"
    if await redis.get(redis_key):
        return  # Skip — updated recently
    await redis.setex(redis_key, LAST_SEEN_TTL, "1")
    await session_repo.upsert_session(
        token_hash=token_hash,
        user_id=user_id,
        workspace_id=workspace_id,
        ip_address=ip,
        user_agent=user_agent,
        db=db,
    )
```

## Open Questions

1. **SAML certificate rotation flow**
   - What we know: SP metadata includes a signing certificate. IdPs cache this.
   - What's unclear: How admins rotate the SAML signing certificate without breaking existing sessions.
   - Recommendation: Implement `/api/v1/auth/saml/metadata` endpoint that always serves current cert. Document that IdP admins must re-fetch SP metadata after rotation.

2. **GoTrue admin API for SAML provider management in self-hosted**
   - What we know: `POST /auth/v1/admin/sso/providers` adds providers; requires `GOTRUE_SAML_ENABLED=true`.
   - What's unclear: Whether the Kong gateway in the project's infra setup already exposes `/auth/v1/admin/*` to the backend service.
   - Recommendation: Verify Kong routing in `infra/supabase/docker/volumes/api/kong.yml` before planning the SAML config API.

3. **OIDC role claim timing**
   - What we know: Supabase OIDC flow issues JWT before our backend can see claims.
   - What's unclear: Whether a post-auth backend hook (webhook from GoTrue) is simpler than frontend-initiated role mapping.
   - Recommendation: Use frontend post-login hook for MVP (simpler); webhook is cleaner but requires GoTrue webhook config.
