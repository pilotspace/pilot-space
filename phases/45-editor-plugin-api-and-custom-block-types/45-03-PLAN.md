Make the editor's PM block types, slash commands, and command palette actions extensible so plugins can register new ones at runtime.

Purpose: Currently PM block types, slash commands, and view zone renderers are hardcoded. This plan opens them up to dynamic registration from plugins while preserving all existing built-in functionality.

Output: Modified editor modules that read from PluginRegistry, plus a bridge hook wiring plugin DOM events to editor operations.
</objective>

  <action>
    1. Modify `frontend/src/features/editor/types.ts`:
       - Keep `PMBlockType` as the built-in union type (no change)
       - Add `type ExtendedPMBlockType = PMBlockType | string` for use in extensible contexts
       - Update `PMBlockMarker` to use `type: ExtendedPMBlockType` (string covers plugin types)

    2. Modify `frontend/src/features/editor/markers/pmBlockMarkers.ts`:
       - Keep `PM_BLOCK_TYPES` array as-is (built-in types)
       - Add `isValidPMBlockType(type: string): boolean` function:
         - Returns true if type is in `PM_BLOCK_TYPE_SET` (built-in) OR if type is in PluginRegistry's registered block types
         - Import `getRegisteredBlockTypes` from PluginRegistry
       - Update `parsePMBlockMarkers` to use `isValidPMBlockType(type)` instead of `PM_BLOCK_TYPE_SET.has(type)` for the validation check
       - This allows plugin-registered block types to be parsed and rendered

    3. Modify `frontend/src/features/editor/view-zones/PMBlockViewZone.tsx`:
       - Keep existing `rendererMap` for built-in types
       - Add fallback for plugin block types: if `type` is not in `rendererMap`, render raw JSON data in a styled `<pre>` block with the plugin's label as header
       - Plugin blocks display their data in a read-only formatted JSON view (no HTML rendering from plugins — all rendering is data-driven, not HTML-driven, to prevent XSS)
       - Update the type annotation to accept `ExtendedPMBlockType` (imported from types.ts)
       - Update `BLOCK_TYPE_LABELS` lookup: check PluginRegistry `getRegisteredBlockTypes()` for label if not found in built-in map

    4. Modify `frontend/src/features/editor/hooks/useMonacoSlashCmd.ts`:
       - Keep `getSlashCommands()` function as-is (built-in commands)
       - In the `useEffect` that registers the CompletionItemProvider, append plugin commands:
         - Import `getRegisteredSlashCommands` from PluginRegistry
         - In `provideCompletionItems`, merge built-in `commands` with `getRegisteredSlashCommands()` entries
         - Plugin commands get `sortText` starting after built-in ones (index + builtInCount)
         - Plugin command `insertText` uses the `trigger` as the PM block marker: e.g., for `/changelog` plugin command with block type `changelog`, insert triple-backtick pm:changelog block with empty JSON
  </action>

  <action>
    Create `frontend/src/features/plugins/integration/usePluginEditorBridge.ts`:
    - Hook: `usePluginEditorBridge(editor: monacoNs.editor.IStandaloneCodeEditor | null, monacoInstance: typeof monacoNs | null)`
    - Listens for DOM CustomEvents dispatched by PluginSandbox handlers:
      a. `plugin:editor-get-content` — reads `editor.getValue()`, returns via event.detail callback
      b. `plugin:editor-insert-block` — reads `{ type, data }` from detail, inserts PM block markdown at cursor position using `editor.executeEdits`
      c. `plugin:editor-replace-selection` — reads `{ text }` from detail, replaces current selection using `editor.executeEdits`
      d. `plugin:block-registered` — triggers editor re-parse of PM blocks (dispatch a model content change event or call the existing decoration refresh)
      e. `plugin:command-registered` — no-op (slash command provider reads from registry on each invocation)
    - Cleanup: removes all event listeners on unmount
    - All event handlers wrapped in try/catch — log errors via console.warn, never throw
    - This hook should be called in EditorLayout or MonacoNoteEditor alongside existing hooks
  </action>
